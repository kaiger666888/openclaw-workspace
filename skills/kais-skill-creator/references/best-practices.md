# Skill 最佳实践指南

## 6. 自动化分级原则（核心设计哲学）

**每个 skill 的每个流程节点，都必须用两个维度评估自动化级别：**

```
           高人工不可替代性
                ↑
    【尊重区】    |    【增强区】
    创意/审美     |    数据分析/参考生成
    人类决策      |    AI辅助决策
                |
← 低错误代价 ———+—— 高错误代价 →
                |
    【自动化区】  |    【半自动区】
    批量/重试     |    需要确认/微调
    AI全自动      |    人机协作
                ↓
           低人工不可替代性
```

### 四区设计规则

**尊重区 — 设置 Human-in-the-loop 卡点：**
- 创意决策、审美判断、价值取向 → 必须暂停等人类确认
- 实现：`<!-- FREEDOM:low -->` + 明确的确认提示
- 示例：skill 发布前的人工审核、内容风格选择

**增强区 — AI 生成结构化数据，人类快速决策：**
- 数据分析、参考生成、方案对比 → AI 出结果，人做最终判断
- 实现：输出 JSON/表格/选项列表，而非最终成品
- 示例：输出 3 个候选方案让用户选择，而非直接执行

**自动化区 — Agent 直接驱动，无需人工确认：**
- 批量操作、重试逻辑、健康检查、格式转换 → 全自动
- 实现：`<!-- FREEDOM:high -->` + 无确认步骤
- 示例：日志轮转、数据格式转换、批量文件处理

**半自动区 — AI 执行 → 人快速二选一 → 继续：**
- 需要微调但有明确选项的操作 → AI 执行主体，人做关键选择
- 实现：`<!-- FREEDOM:medium -->` + Binary Choice
- 示例：AI 生成草稿 → 人选择"发布"或"修改"

### 设计检查清单

创建 skill 时，对每个流程节点自检：

1. **这个节点的错误代价高吗？** → 高代价 → 尊重区或半自动区
2. **这个节点需要人类创意/判断吗？** → 需要 → 尊重区或增强区
3. **这个节点可以批量/重试吗？** → 可以 → 自动化区
4. **有明确的选项让人选择吗？** → 有 → 半自动区

### 在 SKILL.md 中标注

每个流程步骤必须标注所属区域：

```markdown
## 步骤 1：数据分析
<!-- AUTO:自动化区 - 批量处理，无需人工确认 -->
直接执行...

## 步骤 2：方案选择
<!-- SEMI:半自动区 - AI生成3个选项，人选择 -->
...

## 步骤 3：内容创作
<!-- AUGMENT:增强区 - AI生成参考，人做最终决策 -->
...

## 步骤 4：发布确认
<!-- RESPECT:尊重区 - 必须人工确认后才能执行 -->
...
```

## 1. Frontmatter 是触发器

description 字段是 OpenClaw 判断何时激活 skill 的唯一依据。

**正确：**
```yaml
description: "网页抓取和内容提取。当需要抓取网页、提取文章内容时使用。触发词：抓取网页, scrape, extract content"
```

**错误：**
```yaml
description: "A web scraping tool"  # 太模糊
```

## 2. 精简原则

默认假设 AI 已经很聪明。只写它不知道的。
- ❌ 解释什么是 HTTP
- ✅ 说明本 API 的认证方式和特殊参数

## 3. 渐进式披露

```
SKILL.md（核心流程，< 500 行）
  ├── references/api-docs.md（按需加载）
  └── references/examples.md（按需加载）
```

## 4. 自由度分级

| 脆弱度 | 自由度 | 标记 | 示例 |
|--------|--------|------|------|
| 低（格式转换） | 高 | `<!-- FREEDOM:high -->` | "自行选择合适的工具" |
| 中（API调用） | 中 | 默认 | "使用以下命令，可按需调整" |
| 高（发布/删除） | 低 | `<!-- FREEDOM:low -->` | "必须严格按顺序执行" |

## 5. 脚本化原则

- 同一段代码需要重复使用 → 写脚本
- 操作步骤需要确定性保证 → 写脚本
- 依赖版本或环境配置重要 → 写脚本

## 6. 中文优先

- description 包含中英文触发词
- 正文使用中文
- 技术术语保留英文但给出中文对照
- 代码注释使用中文
