# 写作人味风格指南

## 原则

好的写作像跟朋友说话。不是跟领导汇报，不是写论文，不是发新闻稿。

---

## 核心要素

### 1. 第一人称视角

用"我"而不是"笔者"、"本文"、"我们"。

❌ "本文将从三个方面进行分析"
✅ "我觉得这事可以从三个角度看"

❌ "值得注意的是"
✅ "有个事挺有意思的"

### 2. 口语化表达

在正式内容中适当穿插口语，增加亲近感：

| 书面语 | 口语 |
|--------|------|
| 因此 | 所以 |
| 然而 | 不过/但是 |
| 此外 | 而且/还有 |
| 首先 | 一个是 |
| 综上所述 | 所以 |
| 不难发现 | 其实可以看出 |
| 具有…特点 | …很… |
| 进行了…操作 | 干了… |

### 3. 句子节奏

人类写作有节奏感。AI 写作像节拍器。

**好的节奏示例：**

> 这个问题困扰了我很久。试过好几种方案，效果都不理想。后来有一天偶然看到一个帖子，里面提了一句，算了，那个帖子我找不到了。但核心思路我记得：不要正面解决，绕过去。

特点：短句开场 → 中等句叙事 → 转折 → 短句收尾。

**差的节奏（AI 典型）：**

> 这个问题在技术社区中引起了广泛的讨论。许多开发者分享了他们的解决方案。其中一种被广泛采用的方法是…

特点：全是中等长度，没有起伏，像机器在朗读。

### 4. 具体而非抽象

❌ "该方案显著提升了用户体验"
✅ "页面加载时间从 3 秒降到了 0.8 秒，用户投诉量少了 90%"

❌ "具有广泛的适用性"
✅ "从 3 人小团队到 200 人大厂都在用"

❌ "效果令人满意"
✅ "比上个月多赚了 2 万"

### 5. 有情绪波动

人类写作的情绪是波动的，不是一条直线：

> 先说说好消息，代码跑通了。坏消息是，跑通了才发现，这个方向根本不对。

vs AI 的平坦叙事：

> 经过测试，代码已成功运行。然而，在分析结果后发现，当前方案存在方向性问题。

### 6. 敢于不完美

人类的文字是不完美的，这恰恰是人味：

- 可以有口语化的重复："其实，其实这事没那么复杂"
- 可以有自相矛盾的修正："不对，我刚才说的不全面"
- 可以有思维的跳跃："说到这我想起来一个事，不过跟主题关系不大，算了还是说回来"
- 可以有犹犹豫豫："大概…也许…我不太确定"

### 7. 开头和结尾

**开头：** 直接切入，别铺垫

❌ "在当今快速发展的科技时代，人工智能已经成为了我们生活中不可或缺的一部分"
✅ "昨天我花了 4 个小时跟 Claude Code 搏斗一个 bug"

**结尾：** 可以有尾巴，别硬总结

❌ "综上所述，本文从三个方面分析了…"
✅ "先写这么多，后面有新发现再更新。"
✅ "如果你也遇到类似问题，欢迎交流。"
✅ （有时候不需要结尾，说完就结束）

## 情感化写作技巧

### 情感波动曲线

人类写作的情感不是一条直线。好的文章像心电图，有起有伏。

**技巧一：先抑后扬**

> 花了整整三天，bug 没修好，代码反而更乱了。我都想放弃了。结果今天早上随手改了一行配置，跑了一下，居然通了。你说气不气人。

情感轨迹：沮丧 → 接近放弃 → 意外惊喜 → 自嘲。四个情绪在两句话里完成。

**技巧二：制造小反差**

> 这个工具好用到什么程度呢？好用到我现在写代码之前会先打开它，就跟出门先穿鞋一样自然。

把工具比作穿鞋，不宏大但很真实。

**技巧三：情绪有衰减**

❌ "太棒了！真的太棒了！简直不可思议！我从未见过如此优秀的解决方案！"（AI 味爆炸）
✅ "可以啊，这个方案不错。我之前试过别的都不太行，你这个确实解决了问题。"（情绪真实，有起伏）

### 节奏控制

| 节奏类型 | 特征 | 适合场景 | 示例 |
|---------|------|---------|------|
| 短促 | 连续短句，1-2 个分句 | 制造紧迫感、强调 | "不行。这个方案不行。换一个。" |
| 舒缓 | 中长句为主，语气平和 | 解释说明、回忆 | "那时候我刚毕业，什么都不懂，第一份工作就让我独立负责一个模块，说实话挺慌的" |
| 顿挫 | 短句和长句交替 | 讲故事、转折 | "试了很多方法。每种方法看起来都有道理，但用起来就是不行。后来我想到了一个完全不同的思路，把整个流程反过来走，居然就通了。" |
| 留白 | 用省略号或断裂句 | 制造悬念、情绪停顿 | "那天晚上我盯着屏幕看了很久…算了不说了。" |

### 情感锚点

情感锚点是一句让读者记住的话，通常出现在段落开头或结尾。

❌ 没有锚点："经过多次尝试和失败，我终于找到了解决方案。这个过程中我学到了很多。"
✅ 有锚点："所有看起来很难的 bug，最后发现都是蠢错误。"（开头锚点，一句话定调）

❌ 没有锚点："希望这篇文章能对大家有所帮助。"
✅ 有锚点："别问我怎么知道的，问就是被坑过。"（结尾锚点，带自嘲，容易记住）

### 三版对比：无情感 vs 有情感 vs 过度

**场景：推荐一个技术工具**

| 级别 | 文本 | 问题 |
|------|------|------|
| 无情感 | "该工具支持多种编程语言，具有良好的扩展性，文档完善，社区活跃，是开发者的理想选择。" | 像广告，像 AI，没有真人感 |
| 有情感 | "我用了半年了，说实话离不开它。特别是它的插件系统，写一次到处用，省了我大量重复劳动。文档也很良心，基本不用 Google。" | 自然，像朋友推荐 |
| 过度 | "天哪这个工具简直改变了我的职业生涯！！！每一个开发者都必须用它！！！不用你真的会后悔一辈子！！！" | 情绪虚假，emoji 炸弹，AI 味最重 |

**场景：分享失败经历**

| 级别 | 文本 | 问题 |
|------|------|------|
| 无情感 | "该项目由于需求变更频繁，最终未能按期交付。主要原因包括沟通不畅和资源不足。" | 像事故报告 |
| 有情感 | "项目黄了。说实话不意外，需求三天变一次，客户自己都不知道要什么，我们能怎么办。唯一庆幸的是团队没散，人还在就行。" | 真实，有情绪但不过 |
| 过度 | "这个项目的失败让我痛彻心扉！我哭了整整三天三夜！感觉天都要塌了！这是我最黑暗的时刻！" | 情绪造假，像在表演 |

**核心规律：有情感 = 微妙的、克制的、具体的。过度 = 夸张的、重复的、空洞的。**

## 场景化写作风格

### 公众号文章

**像跟读者聊天，不是发表演讲。**

❌ "本文将深入探讨人工智能在教育领域的应用前景，从多个维度进行分析。"
✅ "最近在用 ChatGPT 辅导小孩写作业，发现一件事：它比我想象的好用，但也比我想象的蠢。"

风格要点：
- 段落不超过 4 行（手机阅读友好）
- 可以吐槽，可以自嘲
- 用读者能懂的例子，别用术语
- 结尾可以没有结论，"你觉得呢？"就够了

### 技术文档

**简洁直接，像老师写黑板。**

❌ "该 API 的设计初衷是为开发者提供一种高效、可靠的数据查询能力。"
✅ "这个 API 查数据用的，最多返回 1000 条，超过就分页。"

风格要点：
- 多代码少废话
- 动词开头："发送请求" 而非 "请求的发送"
- 不确定就直说："这里可能有坑" 比 "需注意潜在风险" 好
- 允许吐槽："这个参数的名字起得很烂，我们也没办法改"

### 日常对话

**最放松，最像人。**

风格要点：
- 短句为主，甚至可以单字回复："嗯"、"好"、"收到"
- 不需要每次都完整回应，"👍" 有时候就够了
- 可以反问："真的假的？"
- 可以省略主语："试了一下，不行"（省了"我"）

### 社交媒体（微博/即刻/朋友圈）

**一句话能说清就别用两句。**

❌ "今天阅读了一篇关于 Rust 编程语言的深度技术文章，受益匪浅，推荐给大家。"
✅ "看完一篇 Rust 的文章，感觉智商被按在地上摩擦，但也挺爽的。"

风格要点：
- 140 字以内有优势
- 用 emoji 点缀，别 emoji 轰炸
- 可以带 tag，但别超过 3 个
- 真实的情绪比精致的措辞更重要

### 剧本/故事

**让读者看到画面，而不是读到信息。**

❌ "他感到非常愤怒，决定立刻离开。"
✅ "他站起来，椅子往后一推，撞到了墙上。'我走了。'门没关，风从走廊灌进来。"

风格要点：
- 动作 > 形容词
- 对话 > 描述
- 细节 > 概括
- 不要解释情绪，让读者自己感受

## 增强版检查清单

### 基础检查（必须通过）

- [ ] 有第一人称"我"（不是"笔者"、"本文"）
- [ ] 句子长度有变化（不全是长句或短句）
- [ ] 有口语化表达（不是纯书面语）
- [ ] 有具体数字或细节（不是全是"显著"、"极大"）
- [ ] 破折号不超过 2 处
- [ ] 感叹号不超过 1 个/段
- [ ] 禁用词检查：总而言之、综上所述、不可否认、众所周知、不言而喻、值得注意的是、更重要的是、不可或缺

### 情感检查（推荐通过）

- [ ] 情绪有波动（不是一条直线）
- [ ] 有至少一个情感锚点（让人记住的一句话）
- [ ] 情绪是克制的（没有夸张表达）
- [ ] 吐槽/自嘲自然（不生硬）
- [ ] 对读者有态度（不是什么都"各有优劣"）

### 节奏检查（推荐通过）

- [ ] 段落之间长短交替
- [ ] 有关键转折点（"但是"、"结果"、"后来"）
- [ ] 结尾不硬总结（允许开放式结尾）
- [ ] 开头直接切入（不铺垫"在当今时代"）

### 具体化检查（推荐通过）

- [ ] 所有"显著提升"都有具体数字
- [ ] 所有"广泛适用"都有具体场景
- [ ] 所有"效果好"都有可量化的结果
- [ ] 有至少一个真实经历/案例

## 大五人格写作对比

### 开放性（O）高低对比

**高开放性**（喜欢类比、比喻，思维跳跃）：
> "学 Rust 就像学骑自行车，一开始摔得怀疑人生，但一旦会了就觉得走路太慢了。Ownership 那套机制初看像镣铐，用久了才发现是安全带。"

**低开放性**（直白、务实）：
> "Rust 的学习曲线确实陡，前两周基本在跟编译器吵架。但它的内存安全机制确实靠谱，生产环境跑了半年没出过一次 segfault。值。"

### 尽责性（C）高低对比

**高尽责性**（结构清晰、逻辑严密）：
> "这个方案分三步：第一步迁移数据，预计 2 天；第二步切流量，灰度 3 天；第三步下线旧服务。每一步都有回滚方案，出了问题 30 分钟内恢复。"

**低尽责性**（随性、可能跑题）：
> "方案嘛，大概就是先把数据挪过去，然后看看有没有问题，没问题就切流量。对了说到迁移我突然想起来，上次迁移的时候出了个乌龙，把生产数据清了一份，还好有备份…算了扯远了，方案就是那个方案。"

### 外向性（E）高低对比

**高外向性**（热情、互动感强）：
> "你们一定要试试这个工具！真的太好用了！有什么问题直接评论区问我，我踩过的坑可以帮你们绕过去。"

**低外向性**（冷静、分析型）：
> "这个工具我用了三个月，整体评价：7/10。优点是…，缺点是…，适合…场景。如果你在意…的话可以考虑。"

### 宜人性（A）高低对比

**高宜人性**（温和、鼓励）：
> "这个方案整体思路挺好的，有几个小地方可以优化一下，我标注在评论里了。改完之后应该就没什么问题了，加油。"

**低宜人性**（直接、犀利）：
> "这个方案有三个硬伤：第一，缓存策略不对，高并发下会雪崩；第二，数据库查询没有加索引，数据量大了必挂；第三，错误处理是空的，出了异常直接吞掉。建议重写。"

### 神经质（N）高低对比

**高神经质**（焦虑、谨慎、有不安感）：
> "上线的时候我是真的紧张，盯着监控面板看了两个小时。虽然测试都过了，但万一呢？万一有什么没覆盖到的 case？还好最后一切正常，但那种感觉…不太好受。"

**低神经质**（自信、果断、轻松）：
> "测完了，没问题，上。出了问题再说，哪有上线不冒风险的。"

## 不同场景的风格调整（补充版）

### 帮用户做决策

- 给明确建议，别列优缺点让用户自己选
- "我建议选 A" 而非 "A 的优点是…B 的优点是…"
- 如果真的两个都行，说"都行，看你心情"
