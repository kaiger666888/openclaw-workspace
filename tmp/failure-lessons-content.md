## 🔴 案例1：Cron 报告格式误导性

### 问题描述
定时任务执行报告显示 "Write failed"，但实际上部分任务已成功完成，导致难以判断真实失败原因。

### 根本原因分析
- **表层原因**: Shell 脚本返回值判断逻辑有问题
- **深层原因**: 
  1. 多任务汇总时，单个失败导致整体标记为失败
  2. 缺少细粒度的任务状态追踪
  3. 错误信息未区分「写入失败」与「前置条件不满足」

### 解决方案
1. **状态码分级** - 引入 EXIT_CODES: 0=全部成功, 1=部分失败, 2=全部失败
2. **逐任务日志** - 每个任务输出独立状态行 `[TASK_NAME] ✅/❌`
3. **前置检查分离** - 数据源问题与写入问题分开报告

### 经验教训
- ❌ 不要用单一状态代表批量操作结果
- ✅ 每个子任务都要有可追踪的状态
- ✅ 错误消息要精确到「哪个环节出了什么问题」

## 🔴 案例2：读书笔记 JSON→MD 转换失败

### 问题描述
读书笔记定时任务执行时，JSON 数据转换为 Markdown 格式失败，导致 Notion 页面内容为空。

### 根本原因分析
- **直接原因**: JSON 解析器遇到非标准格式数据崩溃
- **深层原因**:
  1. 数据源格式变更未及时适配
  2. 缺少格式验证层
  3. 无降级策略（fallback）

### 解决方案
1. **防御性解析** - 使用 `jq` 的 `-e` 参数检测格式错误
2. **数据验证** - 转换前检查必需字段是否存在
3. **降级输出** - 解析失败时输出原始数据 + 错误标记
4. **告警机制** - 连续失败 3 次触发通知

### 避坑指南
```bash
# ❌ 危险：直接解析
echo "$json" | jq '.content'

# ✅ 安全：带验证的解析
if echo "$json" | jq -e '.content' > /dev/null 2>&1; then
  content=$(echo "$json" | jq -r '.content')
else
  echo "⚠️ JSON 格式异常，使用降级输出"
  content="数据解析失败，原始内容：$json"
fi
```

## 🔴 案例3：每日英语德语文件未更新

### 问题描述
定时任务连续多天执行，但输出文件停留在 3 月 2 日，看起来像任务没运行。

### 根本原因分析
- **现象**: 文件时间戳未更新
- **可能原因**:
  1. 任务实际未执行（cron 配置问题）
  2. 执行了但内容相同，文件被跳过
  3. 写入路径错误
  4. 权限问题

### 诊断方法
```bash
# 1. 检查 cron 日志
grep "daily-english-german" /var/log/syslog

# 2. 检查文件权限
ls -la /path/to/output/file

# 3. 手动执行看输出
./daily-tasks-v3.sh daily-english-german

# 4. 检查数据源是否更新
curl -I "https://data-source-url"
```

### 解决方案
1. **日志增强** - 每次执行记录时间戳到独立日志文件
2. **状态文件** - 写入 `.last_run` 文件记录执行状态
3. **强制更新** - 即使内容相同也更新时间戳 `touch file`

## 💡 系统性改进建议

### 1. 任务健康度监控
- 每个任务维护 `last_success` 时间戳
- 超过阈值未成功 → 自动告警
- 仪表盘展示任务执行趋势

### 2. 错误分类体系
| 类别 | 代码 | 处理策略 |
|------|------|----------|
| 网络错误 | E_NET | 重试 3 次，间隔递增 |
| 数据格式 | E_FMT | 降级输出 + 告警 |
| API 限制 | E_API | 指数退避等待 |
| 权限错误 | E_PERM | 立即告警，需人工介入 |

### 3. 可观测性三支柱
- **日志**: 结构化 JSON 日志，便于查询
- **指标**: 成功率、耗时、错误分布
- **追踪**: 每次执行生成唯一 trace_id
